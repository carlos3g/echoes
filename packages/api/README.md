## Architecture

The API follows a **module-based architecture** with clear separation of concerns:

### Module Structure

Each domain (quote, author, category, source, tag, user, auth) is a self-contained module with:

```
<domain>/
├── contracts/          # Repository interfaces (Dependency Injection)
├── dtos/              # Request/response Data Transfer Objects
├── entities/          # Domain entities
├── repositories/      # Prisma repository implementations
├── services/          # Business logic services
├── use-cases/         # Application use cases (one per feature)
├── <domain>.controller.ts
└── <domain>.module.ts
```

### Key Patterns

- **Repository Pattern**: All database access through repository contracts (interfaces) with Prisma implementations
- **Use Case Pattern**: Each feature is a separate use case class (e.g., `SignInUseCase`, `FavoriteQuoteUseCase`)
- **Dependency Injection**: Modules provide repositories via contracts, enabling easy testing and mocking
- **Path Aliases**: Use `@app/*` for src imports, `@test/*` for test imports

### Shared Code

Located in `src/shared/`:
- Common DTOs, types, validators
- Utilities used across modules

### Database

- **PostgreSQL** with **Prisma ORM**
- Schema: `prisma/schema.prisma`
- Seeders: `prisma/seeders/`
- Factories: `prisma/factories/`

## Authentication

### Overview

The Echoes API uses a **JWT-based authentication system** with:
- Access and refresh tokens
- Email confirmation for new accounts
- Password reset flow with secure tokens
- Rate limiting on sensitive endpoints

**Technologies:**
- JWT (JSON Web Tokens) for stateless authentication
- Bcrypt for password hashing (10 salt rounds)
- Bearer token scheme for request authorization

### Token Management

**Token Expiration:**
- **Access Token**: 7 days
- **Refresh Token**: 21 days

**JWT Payload:**
```typescript
{
  sub: string;  // User UUID (not database ID)
}
```

**Token Generation:**
1. User UUID stored in JWT `sub` (subject) claim
2. Both tokens signed with `JWT_SECRET` from environment
3. Generated by `AuthService.generateAuthTokens(user)`

### API Endpoints

All authentication endpoints are versioned at `/v1/auth`:

| Method | Endpoint | Auth Required | Rate Limited | Description |
|--------|----------|---------------|--------------|-------------|
| POST | `/sign-in` | No | No | Login with email/password |
| POST | `/sign-up` | No | No | Register new account |
| POST | `/refresh` | No | No | Refresh access token |
| POST | `/forgot-password` | No | Yes (3/2min) | Request password reset |
| POST | `/reset-password/:token` | No | No | Complete password reset |
| GET | `/me` | Yes | No | Get current user |
| PATCH | `/me` | Yes | No | Update user profile |
| PATCH | `/me/avatar` | Yes | No | Update user avatar |
| PATCH | `/change-password` | Yes | No | Change password |
| POST | `/email/confirm/:token` | Yes | No | Confirm email address |
| POST | `/email/resend` | Yes | Yes (3/2min) | Resend confirmation email |

### Authentication Flow

#### 1. Sign Up

**Endpoint:** `POST /v1/auth/sign-up`

**Request:**
```json
{
  "name": "John Doe",
  "email": "john@example.com",
  "username": "johndoe",
  "password": "securepassword",
  "passwordConfirmation": "securepassword",
  "avatar": "<file>"  // Optional, max 2MB, image only
}
```

**Process:**
1. Validate input (password match, email format)
2. Check if email/username already exists
3. Hash password with bcrypt
4. Create user in database
5. Generate email confirmation token (1-day expiration)
6. Send confirmation email
7. Return `null` (user must confirm email before login)

**Avatar Validation:**
- Type: Image files only (`image/*`)
- Max size: 2 MB
- Dimensions validated via custom pipe

#### 2. Sign In

**Endpoint:** `POST /v1/auth/sign-in`

**Request:**
```json
{
  "email": "john@example.com",
  "password": "securepassword"
}
```

**Response:**
```json
{
  "user": {
    "uuid": "...",
    "name": "John Doe",
    "email": "john@example.com",
    "username": "johndoe",
    "emailVerified": true,
    "avatar": "https://..."
  },
  "accessToken": "eyJhbGc...",
  "refreshToken": "eyJhbGc..."
}
```

**Process:**
1. Find user by email (throws 404 if not found)
2. Compare password with stored hash using bcrypt
3. If invalid, throw 401 Unauthorized
4. Generate access & refresh tokens
5. Return user data with tokens

#### 3. Token Refresh

**Endpoint:** `POST /v1/auth/refresh`

**Request:**
```json
{
  "refreshToken": "eyJhbGc..."
}
```

**Response:**
```json
{
  "accessToken": "eyJhbGc...",
  "refreshToken": "eyJhbGc..."
}
```

**Process:**
1. Verify refresh token signature and validity
2. Decode token to extract user UUID
3. Load user from database
4. Generate new access & refresh tokens
5. Return new token pair

#### 4. Accessing Protected Routes

All requests to protected routes must include:

```
Authorization: Bearer <accessToken>
```

The `AuthGuard` automatically:
1. Extracts token from Authorization header
2. Verifies token signature using `JWT_SECRET`
3. Decodes user UUID from token
4. Loads user from database
5. Attaches user to request object (`request.user`)

### Password Reset

Two-step secure password recovery process:

#### Step 1: Request Reset

**Endpoint:** `POST /v1/auth/forgot-password`

**Request:**
```json
{
  "email": "john@example.com"
}
```

**Process:**
1. Find user by email (throws 404 if not found)
2. Delete any existing password reset requests
3. Generate UUID token
4. Hash token with bcrypt before storing
5. Store hashed token in database
6. Send email with reset link containing unhashed token
7. Return `null`

**Email Contains:**
```
Password reset link: {API_DNS}/auth/reset-password/{token}
```

#### Step 2: Complete Reset

**Endpoint:** `POST /v1/auth/reset-password/:token`

**Request:**
```json
{
  "email": "john@example.com",
  "token": "uuid-from-email",
  "password": "newpassword",
  "passwordConfirmation": "newpassword"
}
```

**Process:**
1. Find user by email
2. Find valid password change request (unused, not expired)
3. Compare provided token with stored hash
4. If token invalid, throw 401 Unauthorized
5. Mark request as used (`usedAt` timestamp)
6. Update user password (hashed automatically)
7. Return `null`

**Token Security:**
- Tokens are hashed before storage (bcrypt)
- One-time use: marked with `usedAt` after reset
- Automatic cleanup: background task deletes used requests

### Email Confirmation

Two-phase email verification for account security:

#### Phase 1: Send Confirmation

**Triggered On:**
- User signup
- User requests resend via `/email/resend`

**Process:**
1. Delete all existing confirmation tokens for user
2. Generate new UUID token
3. Hash token with bcrypt before storing
4. Store with 1-day expiration
5. Create confirmation link: `{API_DNS}/auth/email/confirm/{token}`
6. Send email with link
7. User clicks link or submits token

#### Phase 2: Confirm Email

**Endpoint:** `POST /v1/auth/email/confirm/:token`

**Requires:** Bearer token (authenticated user)

**Process:**
1. Validate token:
   - Not yet used (`usedAt` is null)
   - Not expired (`expiresAt` > now)
2. Mark token as used
3. Update user: `emailVerified = true`
4. Send confirmation receipt email
5. Return `null`

**Resend Confirmation:**

**Endpoint:** `POST /v1/auth/email/resend`

**Requires:** Bearer token

**Process:**
- Creates new token
- Invalidates previous tokens
- Sends new confirmation email

### Guards & Decorators

#### AuthGuard

Globally registered guard that executes on every request:

```typescript
@Injectable()
export class AuthGuard {
  // Automatically applied to all routes
}
```

**Flow:**
1. Check if endpoint marked as `@Public()`
2. If public, allow access
3. Otherwise, require valid Bearer token
4. Verify token signature and validity
5. Decode user UUID from token
6. Load user from database
7. Attach user to `request.user`
8. If invalid, throw 401 Unauthorized

#### Decorators

**@Public()**: Marks endpoint as publicly accessible
```typescript
@Public()
@Post('sign-in')
public async signIn() { }
```

**@UserDecorator()**: Injects authenticated user into controller
```typescript
@Get('me')
public async getMe(@UserDecorator() user: User) { }
```

### Security Features

1. **Password Security:**
   - Bcrypt hashing with 10 rounds
   - Passwords never stored in plain text
   - Secure comparison using `bcrypt.compareSync()`

2. **Token Security:**
   - JWT tokens signed with environment secret
   - Separate access and refresh tokens
   - Token verification on every protected request
   - User UUID used (not database ID) for privacy

3. **Temporary Tokens:**
   - Password reset: UUID v4, hashed, one-time use
   - Email confirmation: UUID v4, hashed, 1-day expiration
   - Tokens deleted/marked used after consumption

4. **Rate Limiting:**
   - Forgot password: 3 requests per 2 minutes per IP
   - Email resend: 3 requests per 2 minutes per IP

5. **Email Verification:**
   - New accounts require email confirmation
   - Tokens expire after 1 day
   - Resend capability if missed

6. **Input Validation:**
   - All endpoints use DTOs with class-validator
   - Password minimum 8 characters
   - Email format validation
   - File upload validation (size, type, dimensions)

## Development

### Prerequisites

- Node.js 20+
- PostgreSQL database
- MinIO or S3-compatible storage
- SMTP server (Mailpit for local development)

### Environment Variables

Required in `.env`:

```bash
# API
API_PORT=3000
API_DNS=http://localhost:3000

# JWT
JWT_SECRET=your-secret-key

# Database
DB_HOST=localhost
DB_PORT=5432
DB_DATABASE=echoes
DB_USERNAME=postgres
DB_PASSWORD=password
DB_SCHEMA=echoes

# Email (Mailpit for local dev)
MAIL_HOST=localhost
MAIL_PORT=1025
MAIL_USER=null
MAIL_PASS=null

# Storage (MinIO for local dev)
AWS_ACCESS_KEY_ID=minio
AWS_SECRET_ACCESS_KEY=password
AWS_REGION=us-east-1
AWS_ENDPOINT=http://localhost:9000
```

### Commands

```bash
# Development
yarn dev                # Start dev server with hot-reload

# Database
yarn db:generate        # Generate Prisma client
yarn db:migrate         # Run migrations
yarn db:seed           # Seed database
yarn db:fresh          # Generate + migrate + seed

# Testing
yarn test              # Unit tests (watch mode)
```

## Testing

### Test Structure

```
src/
└── <domain>/
    ├── <domain>.controller.e2e-spec.ts      # E2E tests
    ├── services/<service>.spec.ts           # Unit tests
    └── repositories/<repo>.e2e-spec.ts      # Repository tests
```

### Test Database

E2E tests use a separate test database configured in `.env.testing`:

```bash
DB_DATABASE=echoes_test
DB_SCHEMA=echoes_test
```

Setup test database:
```bash
yarn db:test
```

### Test Utilities

Located in `test/`:
- `server.ts`: Test server setup
- `auth.ts`: Authentication helpers (`getAccessToken`)
- `factories.ts`: Data factories for tests

### Running Tests

```bash
# All E2E tests
yarn test:e2e

# Specific test file
yarn test:e2e auth.controller.e2e-spec

# Unit tests in watch mode
yarn test

# With coverage
yarn test:cov
```

## API Versioning

All endpoints use **URI versioning** with default version `v1`:

```
/v1/auth/sign-in
/v1/quotes
/v1/authors
```

Configure in `main.ts`:
```typescript
app.enableVersioning({
  type: VersioningType.URI,
  defaultVersion: '1',
});
```
